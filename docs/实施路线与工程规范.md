# 实施路线与工程规范

## 1. 文档目标

本文档用于指导 ERP-Go 的分阶段实施、服务脚手架建设、编码规范、测试策略、部署规范和验收标准。当前阶段只完成分析和设计，后续实现应严格遵循本文档和其他设计文档。

## 2. 实施阶段

### 2.1 第一阶段：基础架构和订单履约闭环

目标：跑通“平台订单导入、订单审核、库存锁定、仓库出库、物流发货、平台回传”。

服务范围：

- API 网关。
- 认证权限服务。
- 租户组织服务。
- 商品服务。
- 文件服务。
- 渠道服务。
- 订单服务。
- 库存服务。
- WMS 仓储服务。
- TMS 物流服务。
- 通知服务。
- 基础报表服务。

业务能力：

- 用户、角色、权限、租户。
- 商品 SKU 和平台 SKU 映射。
- 店铺授权和订单导入。
- 订单审核和异常处理。
- 库存余额、库存锁定、释放、扣减。
- 出库单、拣货、复核、打包、发货。
- 物流规则、面单、运单号。
- 发货回传平台。
- 基础销售、库存和出库报表。

验收标准：

- 能创建租户、用户、角色并登录系统。
- 能维护商品和 SKU 映射。
- 能导入或同步平台订单。
- 能审核订单并锁定库存。
- 能通过 WMS 完成出库。
- 能生成或录入运单号。
- 能扣减库存并记录流水。
- 能回传发货状态。

### 2.2 第二阶段：采购、入库和财务结算

目标：打通采购、入库、成本、结算和利润。

服务范围：

- 采购服务。
- 财务服务。
- 报表服务增强。
- 渠道服务结算报告能力。

业务能力：

- 供应商管理。
- 采购计划和采购单。
- 到货通知、收货、质检、上架。
- 采购成本入账。
- 平台结算导入。
- 应收应付。
- 订单利润、SKU 利润、店铺利润。

验收标准：

- 采购单能驱动 WMS 入库。
- 入库能增加库存并写流水。
- 采购和物流成本能进入财务。
- 平台结算能匹配订单。
- 利润报表能按订单、SKU、店铺查询。

### 2.3 第三阶段：智能化和规模化

目标：提升自动化、运营分析和系统扩展能力。

业务能力：

- 自动补货建议。
- 智能分仓。
- 智能物流推荐。
- 多海外仓和 3PL 接入。
- BI 数据仓库。
- 销量预测。
- 多区域部署。
- 租户级数据库隔离。

## 3. 服务脚手架规范

每个服务都应包含：

| 内容 | 说明 |
| --- | --- |
| 服务入口 | 启动 HTTP、gRPC、消息消费者 |
| 配置加载 | 读取环境变量、配置文件、Secret |
| 日志组件 | 输出结构化 JSON 日志 |
| 错误模型 | 统一业务错误码和错误转换 |
| 健康检查 | 存活检查和就绪检查 |
| 指标暴露 | Prometheus 指标 |
| 链路追踪 | OpenTelemetry |
| 数据迁移 | 服务私有 migration |
| 单元测试 | 领域对象和应用服务测试 |
| 接口文档 | OpenAPI 或 gRPC proto |

## 5. 命名规范

- 领域对象名称使用业务语言，避免技术化命名。
- 聚合根以业务名命名，例如销售订单、库存余额、出库单。
- 应用服务以用例命名，例如订单审核应用服务、库存锁定应用服务。
- 仓储接口以聚合名命名，例如订单仓储、库存仓储。
- 外部系统适配器以外部系统名命名，例如 Amazon 适配器、物流商适配器。
- 事件以过去式命名，例如订单已审核、库存已锁定、仓库已出库。

## 6. 编码原则

- 控制器不写业务规则。
- 应用服务负责用例编排和事务边界。
- 领域层不依赖数据库、缓存、消息队列或外部 API。
- 基础设施层实现领域层和应用层定义的接口。
- 业务状态流转必须通过状态机。
- 金额、地址、重量、尺寸等应建模为值对象。
- 库存和财务操作必须有流水。
- 写操作必须支持幂等。
- 外部 API 必须通过适配器访问。

## 7. 数据库规范

- 每个服务独立 migration。
- 禁止跨服务直接访问数据库。
- 所有业务主表包含 `tenant_id`。
- 高频查询字段建立组合索引。
- 外部幂等字段建立唯一索引。
- 大表按时间分区或归档。
- 数据删除优先软删除，财务和库存流水禁止物理删除。

## 8. 接口规范

- 前端统一通过 API 网关访问。
- 服务间高频查询可使用 gRPC。
- 状态变更优先使用领域事件。
- 所有接口传递请求 ID、追踪 ID、租户 ID 和用户 ID。
- 所有写接口支持幂等键。
- 接口变更必须更新文档和契约测试。

## 9. 测试策略

### 9.1 单元测试

重点覆盖：

- 订单状态机。
- 库存锁定、释放、扣减。
- 金额和汇率计算。
- 物流规则匹配。
- 采购质检规则。
- 财务成本分摊。

### 9.2 集成测试

重点覆盖：

- 订单审核到库存锁定。
- 库存锁定到 WMS 出库。
- WMS 出库到库存扣减。
- TMS 面单生成到平台回传。
- 采购到货到库存入库。
- 结算导入到利润核算。

### 9.3 契约测试

重点覆盖：

- 服务间 gRPC 接口。
- 对外 REST API。
- 领域事件 payload。
- 外部平台和物流商模拟接口。

### 9.4 端到端测试

重点覆盖：

- 管理后台订单履约流程。
- 仓库 PDA 拣货、复核、打包流程。
- 采购入库流程。
- 售后退款流程。

### 9.5 性能测试

重点压测：

- 批量订单导入。
- 库存并发锁定。
- 批量出库。
- 物流轨迹同步。
- 报表聚合查询。

## 10. 部署规范

### 10.1 本地开发

- 使用 Docker Compose 启动 PostgreSQL、Redis、消息队列、MinIO、OpenSearch。
- 服务可本地运行，也可容器运行。
- 本地环境应提供外部平台和物流商模拟服务。

### 10.2 测试环境

- 使用 Kubernetes 或 Docker Compose 部署完整链路。
- 每次合并前运行单元测试、集成测试和契约测试。
- 测试环境数据可重置，可自动初始化。

### 10.3 生产环境

- 每个核心服务至少两个副本。
- API 网关、订单服务、库存服务、WMS、TMS 支持水平扩容。
- 数据库定期备份并演练恢复。
- 消息队列开启持久化和死信队列。
- 所有服务接入日志、指标和链路追踪。

## 11. 可观测规范

### 11.1 日志

日志必须包含：

- 服务名。
- 环境。
- 租户 ID。
- 用户 ID。
- 请求 ID。
- 追踪 ID。
- 错误码。
- 业务单号。

敏感字段必须脱敏，例如 Token、密钥、手机号、邮箱、完整地址。

### 11.2 指标

核心指标：

- API QPS、错误率、P95、P99。
- 订单导入量和失败量。
- 库存锁定冲突率。
- 出库及时率。
- 物流轨迹延迟。
- 消息堆积量。
- 数据库慢查询。

### 11.3 链路追踪

- API 网关生成或透传 trace id。
- HTTP、gRPC、消息消费都要接入追踪。
- 外部 API 调用也要生成 span。

## 12. 安全规范

- 用户认证使用访问 Token 和刷新 Token。
- 服务间调用使用内网鉴权、签名或 mTLS。
- 开放 API 必须签名和限流。
- 数据导出必须记录审计。
- 权限变更必须记录审计。
- 店铺授权 Token、物流商密钥、支付密钥必须加密存储。
- 多租户数据访问必须强制带 `tenant_id`。

## 13. 开发验收清单

每个服务完成时至少满足：

- 服务边界和数据归属清晰。
- 有健康检查和配置示例。
- 有数据库 migration。
- 有接口文档。
- 有领域对象单元测试。
- 有关键应用服务测试。
- 有结构化日志。
- 有错误码。
- 有基本指标。
- 有 Dockerfile。
- 有本地启动说明。

## 15. 设计完成后的下一步

建议下一步仍保持设计优先：

1. 补充每个服务的 OpenAPI 或 gRPC 契约草案。
2. 为订单、库存、仓储、物流绘制更细的状态机。
3. 输出数据库 migration 设计稿。
4. 输出服务脚手架模板设计。
5. 确认首期 MVP 的页面清单和接口清单。
