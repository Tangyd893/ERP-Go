# 技术栈与中间件管理规范

## 1. 文档目标

本文档明确 ERP-Go 的基础技术栈约束，避免实现阶段出现后端语言、前端框架和中间件管理方式不统一的问题。

强制规则：

- 后端统一使用 Go + Gin。
- 前端统一使用 Vue 3 + TypeScript + Vite。
- 中间件统一由 `docker` 目录管理。

## 2. 后端技术栈规则

| 项目 | 规则 |
| --- | --- |
| 后端语言 | Go |
| HTTP 框架 | Gin |
| 服务形态 | 微服务 |
| API 协议 | REST、gRPC |
| 异步协作 | RabbitMQ + 领域事件 |
| 可靠消息 | Outbox + Inbox |
| 配置方式 | 环境变量 + 配置文件，生产环境使用 Secret |
| 日志 | 结构化 JSON 日志 |
| 链路追踪 | OpenTelemetry |

实现规则：

- 所有后端业务服务必须使用 Go 实现。
- 首期 HTTP 服务统一使用 Gin。
- 不允许在业务服务中混用 Java、Python、Node.js 等后端运行时。
- 临时脚本、测试工具、数据生成工具可以使用其他语言，但不得成为业务运行链路的一部分。
- Go 服务必须提供健康检查、结构化日志、统一错误码和追踪 ID。
- Go 服务必须把业务逻辑放在应用层和领域层，接口层只做协议转换。
- 领域层不得依赖数据库、缓存、消息队列、HTTP 客户端或外部平台 SDK。

单个服务目录建议：

```text
服务目录/
  cmd/
    server/
      main.go
  internal/
    interfaces/
    app/
    domain/
    infra/
    config/
  api/
  migrations/
  tests/
  Dockerfile
```

## 3. 前端技术栈规则

| 项目 | 规则 |
| --- | --- |
| 前端框架 | Vue 3 |
| 开发语言 | TypeScript |
| 构建工具 | Vite |
| 路由 | Vue Router |
| 状态管理 | Pinia |
| UI 组件 | Element Plus |
| API 客户端 | 统一封装 HTTP client |
| 代码质量 | ESLint、Prettier、TypeScript 类型检查 |

前端应用划分：

```text
frontend/
  admin-web/
  warehouse-pda/
  dashboard-web/
  shared/
```

实现规则：

- 所有前端业务应用必须使用 Vue 3 + TypeScript + Vite。
- 不使用 React、Angular 或其他前端框架实现业务应用。
- 页面路由统一由 Vue Router 管理。
- 跨页面状态统一由 Pinia 管理。
- API 调用必须通过共享 API client，不在页面组件中直接拼接请求。
- 权限控制必须有路由级、菜单级、按钮级和数据级约束。
- 表格、筛选、分页、导入导出、弹窗、详情页布局应沉淀为共享组件。
- WMS PDA 页面必须优先考虑扫码、单手操作、弱网重试和重复提交幂等。

## 4. 中间件 Docker 管理规则

所有本地开发和测试依赖的中间件统一放在 `docker` 目录中管理。开发者不应依赖本机裸装 PostgreSQL、Redis、RabbitMQ、MinIO 或 OpenSearch。

`docker` 目录建议：

```text
docker/
  compose/
    docker-compose.dev.yml
    docker-compose.test.yml
  images/
  nginx/
  k8s/
  observability/
  scripts/
```

首期中间件清单：

| 中间件 | 用途 | 本地管理方式 |
| --- | --- | --- |
| PostgreSQL | 业务数据库 | Docker Compose |
| Redis | 缓存、分布式锁、短期状态 | Docker Compose |
| RabbitMQ | 领域事件、异步任务、死信队列 | Docker Compose |
| MinIO | 商品图片、面单、发票、导入导出文件 | Docker Compose |
| OpenSearch | 订单、商品、物流轨迹、日志检索 | Docker Compose |
| Prometheus | 指标采集 | Docker Compose |
| Grafana | 指标看板 | Docker Compose |
| Loki | 日志聚合 | Docker Compose |
| Tempo 或 Jaeger | 链路追踪 | Docker Compose |

Docker Compose 规则：

- `docker/compose/docker-compose.dev.yml` 管理本地开发中间件。
- `docker/compose/docker-compose.test.yml` 管理测试环境中间件。
- Compose 文件必须声明网络、数据卷、健康检查和默认环境变量。
- 默认账号密码只允许用于本地开发和测试，不得用于生产环境。
- 中间件数据卷、日志、临时文件不得提交到 Git。
- 端口映射必须在文档中说明，避免与前后端服务冲突。
- 健康检查失败时，业务服务不应标记为 ready。

端口建议：

| 中间件 | 端口 |
| --- | --- |
| PostgreSQL | `5432` |
| Redis | `6379` |
| RabbitMQ | `5672` |
| RabbitMQ Management | `15672` |
| MinIO API | `9000` |
| MinIO Console | `9001` |
| OpenSearch | `9200` |
| Prometheus | `9090` |
| Grafana | `3001` |
| Jaeger | `16686` |

## 5. 技术选型变更规则

以下变更必须通过架构决策记录确认：

- 后端语言从 Go 替换为其他语言。
- Go HTTP 框架从 Gin 替换为其他框架。
- 前端框架从 Vue 替换为其他框架。
- 消息队列从 RabbitMQ 替换为 Kafka、NATS 或其他系统。
- 数据库从 PostgreSQL 替换为 MySQL 或其他数据库。
- 对象存储从 MinIO/S3 兼容方案替换为其他方案。
- 本地中间件不再由 Docker Compose 管理。

架构决策记录必须说明背景、决策、备选方案、影响范围、迁移成本和回滚方案。
