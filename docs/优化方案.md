# ERP-Go 项目状态分析与优化方案

## 1. 分析口径

本方案基于 2026-05-22 对仓库 `D:\workspace\coding\ERP-Go` 的本地检查、文档阅读和命令验证整理。检查开始时仓库位于 `main` 分支，工作树干净。

已执行的验证：

- `go test ./...`：通过。
- `go vet ./...`：通过。
- `go build ./...`：退出码为 0，但 Go 尝试写入 `D:\Document\gopath\pkg\mod\cache` 时出现 `Access is denied` 的 stat cache 警告，建议后续把 `GOPATH/GOCACHE` 固定到仓库内或 CI 可写目录。
- 前端类型检查：直接调用本地 `vue-tsc` 时，`warehouse-pda` 和 `dashboard-web` 通过，`admin-web` 因无法解析 `@erp/shared` 失败。
- 前端构建：直接调用本地 `vite build` 时，`warehouse-pda` 和 `dashboard-web` 均因缺少 `@rollup/rollup-win32-x64-msvc` 失败。
- 当前 shell 可找到 Node，但找不到 `npm` 命令；这会影响开发者按 README 执行 `npm run ...`。

## 2. 当前状态判断

项目已经从“纯设计和目录骨架”推进到“可编译的微服务 MVP 骨架”。后端 13 个规划服务均有 `cmd/server`、`internal/app` 和 HTTP 层；除 `report-service` 作为聚合查询服务暂未设置仓储外，其余 12 个服务均有 repository 和服务级 migration。API Gateway 已覆盖 13 个服务代理路由，并支持通过环境变量覆盖服务目标地址。

当前主要矛盾已经不是“缺少服务目录”，而是“主业务闭环尚未被真实接口、事件消费、集成测试和前端页面完整打通”。也就是说，项目的骨架密度已经足够，下一阶段要收敛到可演示、可验证、可回归的订单履约闭环。

## 3. 关键现状快照

| 维度 | 现状 | 主要问题 |
| --- | --- | --- |
| 后端服务 | 13 个服务目录完整，12 个服务具备仓储层，Gateway 路由覆盖全部规划服务 | 多数服务仍偏 CRUD/骨架，跨服务用例和补偿流程不足 |
| 事件与流程 | `shared/events`、`shared/outbox`、`shared/workflows` 已建立，P4 出库流程有 5 个测试 | 仍使用 `LogPublisher`，RabbitMQ publisher/consumer、Worker、死信和重试未落地 |
| 数据迁移 | 服务级 migration 12 个 SQL、约 53 张表；根级 migration 16 个 SQL、约 26 张表 | 根级与服务级迁移双轨并存，容易产生表结构漂移 |
| 测试 | 后端 20 个测试函数：Inventory 9 个、Order 6 个、P4 workflow 5 个 | IAM、Tenant、Product、Channel、Warehouse、Transport、Finance 等核心包缺少测试 |
| 前端 | `admin-web` 18 个视图，`warehouse-pda` 2 个视图，`dashboard-web` 1 个基础页 | `admin-web` 类型解析失败；多处页面仍使用 mock 数据；Vite 构建受 Rollup 可选依赖阻塞 |
| 工程脚本 | Makefile 和 README 已提供常用命令 | Windows 环境下 `npm` 缺失、Makefile 中 Unix 风格命令较多，跨平台体验不稳定 |
| 部署运维 | Docker Compose 中间件齐全，K8s 有 Gateway 基础 yaml | 生产级 Helm/K8s、Secret、备份、告警、链路追踪仍未成体系 |
| 文档 | 架构、领域、接口、数据、里程碑文档较完整 | README 与里程碑文档的完成度描述存在不一致，需要统一为单一事实源 |

## 4. 优先级路线

### P0：先恢复可验证基线

目标：让所有开发者能在同一套命令下稳定验证项目，避免后续业务开发建立在不稳定环境上。

1. 修复前端依赖与命令环境。
   - 明确 Node/npm 版本要求，保证 `npm` 在开发 shell 可用。
   - 重新安装或补齐 Windows Rollup 可选依赖。
   - 修复 `admin-web` 对 `@erp/shared` 的类型解析，可在 `tsconfig` 和 Vite alias 中显式声明共享包路径，或为 `@erp/shared` 建立稳定的 build/types 输出。
   - 验收：`npm run typecheck`、`npm run build:admin`、`npm run build:pda`、`npm run build:dashboard` 全部通过。

2. 统一 migration 策略。
   - 建议以 `backend/services/*/migrations` 作为业务表唯一事实源。
   - 根级 `backend/migrations` 只保留全局基础设施迁移，或迁入 `docs/archive` 标注为历史设计稿。
   - 明确 outbox/inbox 表归属：如果作为共享基础设施，可单独维护在 `backend/migrations/outbox`，并在启动文档里说明执行顺序。
   - 验收：新增表结构只允许出现在一个迁移入口；README、里程碑和数据模型文档统一描述。

3. 固化后端验证脚本。
   - 设置仓库内 `GOCACHE`，避免访问用户目录缓存导致权限警告。
   - 为 Windows/PowerShell 增加等价命令，避免 Makefile 中 `export`、`grep`、`awk`、`rm` 等 Unix 假设成为阻塞。
   - 验收：`go test ./...`、`go vet ./...`、`go build ./...` 在干净环境无权限警告。

4. 修正事件基础细节。
   - 检查 `OrderAppService.emitEvent` 的参数顺序和 `AggregateID` 写入，目前调用处存在把 `tenantID` 和 `orderID` 语义混淆的风险。
   - 将 HTTP handler 中的 `context.Background()` 替换为请求上下文，避免取消、超时和 trace 断链。
   - 验收：为订单创建、审核、取消事件补单元测试，断言 event type、aggregate id、payload 均正确。

### P1：打透首期订单履约闭环

目标：用真实后端接口和最小前端页面跑通“订单审核 -> 库存锁定 -> 出库单 -> 出库 -> 库存扣减 -> 订单发货 -> 平台回传”。

1. 事件从日志发布升级到 RabbitMQ。
   - 实现 RabbitMQ publisher/consumer。
   - Outbox polling 投递成功后更新状态，失败进入重试。
   - 引入 Inbox 消费幂等记录。
   - 验收：同一事件重复投递不会重复锁库存或重复创建出库单。

2. 完成 P4 Worker 和补偿路径。
   - 订单审核成功消费后锁定库存。
   - 库存锁定成功后创建出库单。
   - 仓库出库后触发库存扣减和订单状态更新。
   - 回传失败进入重试队列，保留人工补偿入口。
   - 验收：后端集成测试覆盖成功、库存不足、重复事件、回传失败四类场景。

3. 收敛前端 MVP。
   - 后台先连接登录、订单、库存、出库、物流配置的真实 API。
   - PDA 先完成任务列表、拣货、复核、打包、出库确认。
   - mock 数据只能保留在显式 demo 模式，默认环境不应悄悄回退 mock。
   - 验收：业务人员无需调用接口即可完成首期主流程。

### P2：补齐质量和业务纵深

目标：让系统具备试运行条件，而不是只能演示 happy path。

1. 测试加固。
   - IAM：登录、刷新、权限、角色分配、审计。
   - Tenant：组织、部门、岗位、租户隔离。
   - Product/Channel：SKU 映射、订单导入幂等。
   - Inventory/Warehouse/Transport：并发锁定、出库扣减、面单幂等。
   - Finance：金额、汇率、成本分摊、利润计算。

2. 契约与接口治理。
   - 为 REST/gRPC/事件 payload 建立契约测试。
   - 字段新增必须兼容旧消费者；删除、改名、语义改变必须升版本。
   - Gateway 路由、鉴权跳过列表和错误响应需要有回归测试。

3. 采购入库和财务结算。
   - 采购单审核 -> 到货通知 -> WMS 入库 -> 库存增加。
   - 平台结算导入 -> 成本归集 -> 订单利润。
   - 库存和财务都必须保留流水、对账和冲销能力。

4. 可观测与运维。
   - 统一日志字段：service、tenant_id、user_id、request_id、trace_id、business_no。
   - 接入 Prometheus 指标、慢查询、消息堆积、死信数量、出库及时率。
   - 补齐 K8s/Helm、Secret、备份恢复、告警规则和运维手册。

## 5. 建议的 30 天节奏

| 时间 | 目标 | 交付物 |
| --- | --- | --- |
| 第 1 周 | 绿色验证基线 | 前端 typecheck/build 通过；Go build 无缓存权限警告；migration 单一事实源；事件 aggregate id 修复 |
| 第 2 周 | 后端 P4 闭环 | RabbitMQ publisher/consumer；订单审核到出库单创建集成测试；库存扣减和订单发货事件 |
| 第 3 周 | 前端 MVP 接口化 | admin 订单/库存/出库页面接真实 API；PDA 拣货/复核/打包可用；mock 默认关闭 |
| 第 4 周 | 试运行加固 | 端到端烟测；异常重试和补偿；核心指标和日志；README 与里程碑同步更新 |

## 6. 验收清单

- [ ] `go test ./...`、`go vet ./...`、`go build ./...` 在本地和 CI 稳定通过。
- [ ] `npm run typecheck` 和三个前端构建全部通过。
- [ ] migration 入口统一，根级和服务级 SQL 不再双轨维护同一业务表。
- [ ] 订单审核事件的 aggregate id、tenant id、payload 语义正确且有测试覆盖。
- [ ] RabbitMQ 发布、消费、重试、死信、Inbox 幂等可用。
- [ ] 首期订单履约主链路有后端集成测试和前端端到端烟测。
- [ ] 默认业务页面不依赖 mock 数据；mock 只在显式 demo/dev 模式启用。
- [ ] README、里程碑、优化方案三处对项目状态描述一致。
- [ ] 库存和财务关键写操作都有流水、幂等键、审计和对账方案。
