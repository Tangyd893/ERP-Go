## 2. 总体完成度判断

项目已经越过“目录骨架阶段”，进入“可编译、可测试、可局部运行的微服务 MVP 中段”。工程底座、领域模型、服务分层、基础迁移、事件基础设施和测试基线已经明显成型；真正限制交付的重点已经转向真实业务闭环、前端可操作流程、集成测试、生产化部署和本地/CI 可重复验证。

按里程碑清单统计，当前已勾选 104 项、未完成 141 项，清单完成率约 42.4%。结合代码事实判断，整体完成度可估算为 **40% - 45%**：

- 工程与架构底座：完成度高，约 85% - 90%。
- 后端领域与服务骨架：完成度较高，约 65% - 75%。
- 首期订单履约闭环：Order 侧 Outbox + P4 编排已跑通雏形，缺 PDA、独立 Consumer、E2E，约 50% - 55%。
- 前端业务可用性：页面与 Store 已铺开，PDA 为静态占位，约 35% - 45%。
- 采购、财务、报表纵深：领域模型和部分接口已有，跨服务闭环仍早期，约 20% - 30%。
- 自动化验证、端到端、生产部署：仍是主要短板，约 20% - 35%。

一句话结论：**这是一个工程骨架扎实、领域建模较全，但尚未达到试运行条件的 ERP MVP。下一阶段不应继续横向铺服务，而应集中打透“订单履约 + PDA 作业 + 异常补偿 + E2E 验证”这条主链路。**

## 5. 优化优先级

### P0：固定可重复验证基线

目标：让本地和 CI 都能稳定跑出同一组结论，先把“能否验证”从业务开发中拆出来。

1. 建立 CI 与统一验证入口。（**已完成基础版**）
   - GitHub Actions：`.github/workflows/ci.yml`，push/PR 时执行 `scripts/verify.sh`。
   - 本地入口：`scripts/verify.ps1`（Windows）、`scripts/verify.sh`（Linux/macOS）、`make verify`。
   - `.gitignore` 已修正：`/server` 替代裸 `server`，避免误忽略 `cmd/server/`。
   - 待办：在 npm 不可用的 Windows 环境复验前端；CI 首次 push 后确认绿灯。

2. 把前端验证纳入可执行事实源。
   - 修复或记录 `npm` 不可用、`node.exe` 拒绝访问、`.bin` shim 拒绝访问的问题。
   - 若继续使用 npm workspaces，确保 `npm run typecheck` 和三个 build 可在干净 Windows 环境复现。
   - 验收：不再只依赖 README 描述；有最近一次可复现的构建记录。

3. 引入迁移执行与回滚规范。
   - 保持服务级 migration 为业务表唯一事实源；补齐 `report-service` 迁移或文档明确其为无状态聚合服务。
   - 引入 `golang-migrate` 或同等工具，明确 outbox 基础设施迁移顺序。
   - 验收：新环境可一键创建数据库结构，迁移版本可查询、可回滚。

### P1：把首期订单履约链路跑实

目标：用真实接口跑通“订单审核 -> 库存锁定 -> 出库单 -> PDA 作业 -> 出库 -> 库存扣减 -> 订单发货 -> 回传/补偿”。

1. 明确事件架构并完成 Consumer 落地。
   - **现状**：Order 已在进程内做 Outbox 轮询 + P4 handler，不宜重复建设第二套轮询。
   - **待做**：决策是否以 RabbitMQ Consumer 替代/补充进程内轮询；若保留双通道，需定义职责边界。
   - 将 `RabbitMQConsumer` 接入可运行 worker（或确认仅 Outbox 轮询即可满足首期）。
   - 明确队列、binding key、重试、死信和人工补偿入口。
   - 验收：重复事件、库存不足、出库失败、状态更新失败均有集成测试。

2. 补齐 WMS PDA 最小可用流程。
   - PDA 支持任务列表、拣货扫码、复核扫码、打包、称重、出库确认（替换 `Home.vue` 静态占位）。
   - 扫码必须支持幂等和异常上报。
   - 验收：仓库人员不直接调 API 即可完成一单出库。

3. 收敛 admin 订单、库存、出库、物流页面。
   - 移除 `NotificationCenter` 等隐式 mock；demo 模式需显式环境变量或配置开关。
   - 页面错误、超时、空态、权限不足需有明确反馈。
   - 验收：业务人员能在后台查看履约状态、库存流水、出库单、物流和异常任务。

### P2：补齐服务真实业务深度

目标：把“可编译四层结构”升级为“真实业务接口可回归”。

1. 为 IAM/Tenant/Product/Channel/Warehouse/Transport/Purchase/Finance 补 app 与 HTTP 集成测试。
2. 为 repository 层增加数据库集成测试，优先覆盖库存、订单、出库、采购、财务流水。
3. 明确 fallback/占位响应边界：开发降级可保留，生产与验收环境须显式失败或 `degraded` 标识。
4. 扩展契约测试：Gateway 路由已有 4 个，需补各服务 HTTP 响应、事件 payload 契约；`testing/contract` 目录应落地真实用例。
5. 为 `shared/outbox` 补单元测试；推进 `file-service` 与 MinIO 对象存储接入。

### P3：扩展采购入库与财务闭环

目标：在订单履约稳定后，形成库存资金账闭环。

1. 将 `HandleInboundReceived` 注册到 purchase/warehouse 或 inventory 服务，打通：采购单审核 -> 到货 -> WMS 入库 -> 库存增加 -> 流水。
2. 平台结算导入 -> 订单匹配 -> 成本归集 -> 利润计算；`report-service` 改为真实聚合查询而非硬编码。
3. 财务调整、库存调整、取消/退货须保留流水、审计、冲销和对账能力。

### P4：生产化与试运行准备

目标：从“本地可演示”推进到“可部署、可观测、可恢复、可移交”。

1. 补齐 Helm/K8s、Secret、配置分层、资源限制、健康检查。
2. 建立 PostgreSQL、RabbitMQ、MinIO 的备份恢复演练。
3. 接入结构化日志、Prometheus 指标、链路追踪、慢查询、消息堆积和死信告警。
4. 输出部署手册、运维手册、用户操作手册和试运行验收清单。
